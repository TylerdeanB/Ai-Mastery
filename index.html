Module 01 — LLM Fundamentals

How Language Models Actually Work

No fluff. What you need to understand about the technology you're working with — and how to get more from it.

What Is an LLM?

LLM — Large Language Model
A statistical prediction machine trained on billions of text examples. Given any input text, it predicts the most likely next word — over and over — until it produces a response. It doesn't "think." It pattern-matches at massive scale. The intelligence is emergent from that scale.
Analogy: Autocomplete on your phone, but trained on essentially all human text ever written, producing surprisingly sophisticated output.

Critical Terminology

Token
The unit of text an LLM processes. Roughly 1 token ≈ 4 characters ≈ 0.75 words. All LLM limits — speed, cost, context — are measured in tokens. This is what you pay for when using an AI API.
Example: The word "hospitality" = 4 tokens. "Hi" = 1 token. Every character in your prompt and the AI's response costs tokens.
Context Window
The maximum amount of text (in tokens) an LLM can "see" at once — both input AND output combined. Think of it as short-term memory. Once exceeded, earlier content gets cut off. Claude Sonnet has a 200,000 token context window — one of the largest available.
Practical meaning: You could paste an entire book into Claude's context and it could analyse the whole thing at once.
Temperature
Controls creativity vs consistency. 0.0 = always the same deterministic answer. 1.0 = creative and varied. For business data analysis: use 0.1–0.3. For creative writing: 0.7–0.9.
Rule of thumb: low temperature for analysis and facts, high temperature for brainstorming and creativity.
System Prompt
Instructions given to the model before the conversation starts. Sets the role, rules, and behaviour. The end user never sees it. This is the primary lever for controlling AI behaviour in any product you build.
Example: "You are a financial analyst. Always respond in JSON. Never speculate without data." — This shapes every response that follows.
RAG — Retrieval Augmented Generation
Instead of training a model on your data, you retrieve live data at query time and inject it into the prompt. No fine-tuning needed. This is how most production AI applications work — the model is generic, the data is specific.
Example: A customer service bot that pulls your actual product catalogue into Claude's context before answering questions. The model doesn't "know" your products — it reads them fresh each time.
Inference
The act of running a prompt through an LLM to get a response. Every AI answer = one inference call. In production systems, inference is what you pay for — per token, per call.
Hallucination
When an LLM confidently generates false or made-up information. It's a fundamental property of how LLMs work — they predict plausible text, not guaranteed truth. The best models minimise it but can't eliminate it entirely.
Mitigation: Ground AI responses in real injected data rather than asking the model to recall facts from training. Data-grounded prompts reduce hallucination risk significantly.
Fine-tuning
Taking a base LLM and training it further on your own data to specialise its behaviour. Expensive and complex. In most cases, well-designed prompting (prompt engineering) achieves 90% of the result at zero extra cost.

Prompting Techniques — Your Most Valuable Skill

Technique 01

Role Assignment

"You are a senior financial analyst specialising in operational efficiency." This shapes everything that follows. The model performs better with a specific identity and context.

Technique 02

Output Format Specification

"Respond in JSON with keys: insights[], severity (info|warn|danger), action_required (bool)." Exact format = reliable, parseable, programmable output every time.

Technique 03

Chain of Thought

"Before answering, reason through the problem step by step." Forces reasoning before conclusions — dramatically improves accuracy on complex multi-variable tasks.

Technique 04

Few-Shot Examples

Show 2-3 ideal input/output examples before your real request. "Here are examples of good output: [x, y]. Now generate for this input:" The model calibrates to your exact standard.

Technique 05

Negative Constraints

"Do NOT include any preamble. Do NOT explain your reasoning. Output ONLY valid JSON." Tell the model what NOT to do — just as important as what to do.

Technique 06

XML Tags for Structure

Wrap sections in XML tags: <data>...</data> <rules>...</rules>. Claude responds well to structured XML prompts — especially useful for long, complex system prompts.

Example — Well-Structured System Prompt
// System Prompt (hidden from end user) SYSTEM: "You are a business performance analyst. <rules> - Analyse only the data provided — do not speculate. - Flag any metric more than 10% outside its target as a warning. - Output ONLY valid JSON. No prose. No markdown. - Order insights by severity: danger first, then warn, then info. </rules>" // User Message (contains live injected data) USER: "<data>[your business data here]</data> Generate up to 5 insights ordered by priority." // Model returns structured, predictable output { "insights": [ { "type": "cost", "severity": "danger", "message": "Labour 18% over target — immediate action needed" } ]}

Key Terminology

Tool Use / Function Calling
The ability to give an LLM access to external functions it can call during a conversation — search the web, query a database, call an API, run code. This is what makes AI useful in production. It can interact with the real world, not just its training data.
Example: A customer support AI that can look up actual order status in your database before answering. Without tool use, it would have to guess.
MCP — Model Context Protocol
Anthropic's open standard for connecting AI models to external data sources and tools. A universal plug. Instead of every developer writing custom integrations, MCP provides a standard interface that any compatible tool can plug into.
Agent Loop
The process an agentic AI follows: (1) Observe current state → (2) Plan next action → (3) Execute via tool → (4) Observe result → (5) Done or loop again. The human defines the goal. The agent figures out the steps.
Autonomous vs Supervised Agents
Autonomous agents complete tasks without checking back. Supervised agents pause at key decisions for human approval. For any system that touches real money, data, or communications — supervised is safer. Build trust before granting autonomy.
Orchestrator Agent
In a multi-agent system, the orchestrator coordinates specialised agents, receives their outputs, resolves conflicts, and delivers a unified result. The "manager" in a team of AI specialists.

Real-World Agentic AI Examples

Level 3 — Tool-Using

AI Customer Support

AI receives a question → calls your order database → reads actual order status → gives an accurate, real-time answer. Not guessing from training data — reading live data via a tool.

Level 4 — Agentic

AI Research Assistant

Given "write a competitive analysis of the top 5 CRMs," the agent: searches the web, reads product pages, pulls pricing, reads reviews, synthesises into a report. Multiple tool calls, no human intervention.

Level 5 — Multi-Agent

AI Operations Platform

Separate specialised agents for finance, marketing, HR, and ops — each monitoring their domain. An Orchestrator Agent synthesises their daily outputs into an executive brief. The future of business intelligence.

⚠
The Autonomy TrapThe temptation is to make agents fully autonomous as fast as possible. Resist it. Supervised agents that ask for approval at key decisions build trust, catch errors, and stay safe. Give agents more autonomy incrementally as they prove reliability. Move fast on capability, slow on autonomy.

Module 05 — Automation & Workflows

Making Your Systems Run Without You

AI without automation is a one-time trick. AI with automation is a business. This module covers how to build workflows that run 24/7 with zero manual input.

AI Automation vs Traditional Automation

Traditional Automation

Rule-Based

If X happens → do Y. Rigid. Can't handle edge cases or varied input formats. Good for: sending emails when a form is filled, copying files on a schedule, simple trigger/action chains.

AI Automation

Intelligence-Based

Understands context, handles exceptions, adapts to varied input. Analyses complex multi-variable data and tells you what to act on. The nuance is AI's advantage over rigid rules.

⚡
Use Both TogetherTraditional automation handles the scheduled, mechanical work (pull data, transform format, push to API). AI handles the analysis layer (given this data, what matters and why). They stack. Don't choose — combine them.

Core Workflow Patterns

Trigger → Action
The simplest pattern. An event triggers a response. New file uploaded → process it. New customer signs up → send welcome email. The foundation of all automation — n8n, Make, and Zapier are all built on this pattern.
Polling Loop
Check for new data on a schedule. Every hour, check if there's new sales data. Every night, pull tomorrow's bookings. Reliable and simple — but not real-time.
Webhook
Push-based instead of pull-based. Instead of you checking every hour, the source system notifies you the instant something changes. Faster, more efficient than polling. Requires the source system to support webhooks.
Example: Instead of checking for new orders every 30 minutes, your e-commerce platform sends a webhook to your system the moment an order is placed. Instant, zero wasted polling calls.
Data Transform
Converting raw data from its source format into your system's standard schema. Every external system outputs data differently — transforms are the glue that standardises everything before analysis.
Example: A CRM exports contacts as CSV with column "First_Name". Your system expects "firstName". A transform renames the column. Simple but critical for reliable data pipelines.

Module 06 — Sales & Positioning

How to Sell AI to Non-Technical Buyers

You could understand AI better than anyone in the room and still lose to someone who communicates it better. This module fixes that.

🎯
The Fundamental RuleSell outcomes. Not technology. A buyer doesn't care what model you use, what the context window is, or how many API calls it makes. They care what problem it solves, how much it costs, and whether it's worth it. Start there. Always.

How to Talk About AI Without Losing People

Don't Say

"It uses RAG with Claude Sonnet at temperature 0.1"

Technical jargon in a sales context signals you haven't thought about your buyer. It creates distance, not trust.

Do Say

"It reads your live data and tells you what to act on — automatically"

Same system, different frame. Outcome-first language. The buyer understands the value immediately.

Don't Say

"It's a multi-agent Level 4 agentic system"

Meaningless to a non-technical buyer. Worse — it sounds like you're trying to impress them rather than help them.

Do Say

"It monitors your operations 24/7 and flags problems before they cost you money"

Clear, concrete, valuable. They can immediately picture what this does for them.

Common Objections — How to Handle Them

"We already have reports from our existing systems"
Reports are backwards-looking. AI analysis is real-time. A report tells you what happened last night. An automated AI system tells you what's happening right now and what to do about it. One is a post-mortem. One is a command centre.
"How accurate is the AI?"
The AI doesn't guess. It analyses your actual data from your actual systems and applies rules you've defined. It's pattern detection on your own numbers — not speculation. If the data is accurate, the analysis is accurate.
"Is my data secure?"
The AI reads your data in-session for analysis — it doesn't store it separately or use it to train other models. Claude's API has enterprise-grade data handling. Your data stays yours.
"It's expensive"
Reframe as ROI. The question isn't whether it costs money — it's whether it returns more than it costs. One problem caught, one decision improved, one cost avoided — does that exceed the monthly price? In most cases, yes, many times over.
